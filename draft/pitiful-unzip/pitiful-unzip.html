О бедном Unzip'е замолвите слово...

[2017, Ноябрь]

Как-то раз (когда я экспериментировал с umask) я наткнулся на такой вопрос: Unzipping file whilst getting correct permissions?. Только один пользователь, David Tanzer, ответил офигенно правильно и подробно: zip-файлы могут хранить права доступа. В общем, я решил полазить по Сети и собрать разрозненную информацию в одной ламерской заметке.

Исследования буду проводить в LinuxMint 17.1. Для начала узнаем, какие именно программы установлены: 

yura@Aspire-ES1-521 / $ which zip
/usr/bin/zip

yura@Aspire-ES1-521 / $ which unzip
/usr/bin/unzip

И их версии: 

yura@Aspire-ES1-521 / $ zip -v
...
This is Zip 3.0 (July 5th 2008), by Info-ZIP.
...

yura@Aspire-ES1-521 / $ unzip -v
UnZip 6.00 of 20 April 2009, by Debian. Original by Info-ZIP.
...

И наконец, их происхождение: 

yura@Aspire-ES1-521 / $ dpkg --search /usr/bin/zip
zip: /usr/bin/zip

yura@Aspire-ES1-521 / $ dpkg --search /usr/bin/unzip
unzip: /usr/bin/unzip

Как видим, Zip и Unzip - не часть coreutils, а самостоятельные пакеты (по крайней мере, в LinuxMint).

Сжимаем-разжимаем

Создадим в /tmp каталог zip_archive_test. А внутри него построим такую структуру: 

zip_archive_test/:
dir  file.txt

zip_archive_test/dir:
file1.txt  file2.txt

Назначим еще необычных владельцев (таких у меня в системе нет) и странные права доступа (тут главное не переусердствовать, особенно с правами доступа к каталогам), чтобы они не путались с "обычными" файлами [2]. Еще для файла file.txt назначим биты setuid, setgid и sticky-бит. И назначим эти же биты для директории dir. В итоге получится вот что (c помощью опции --almost-all опустим ненужные каталоги . и ..): 

yura@Aspire-ES1-521 /tmp $ sudo ls -l --almost-all --recursive zip_archive_test/
zip_archive_test/:
total 4
drwsr-srwt 2 1050 1051 4096 Nov 20 20:48 dir
-rwSr-S--t 1 1025 1026    0 Nov 20 20:47 file.txt

zip_archive_test/dir:
total 0
-rw-r----- 1 1050 1051 0 Nov 20 20:48 file1.txt
-rw-r----- 1 1050 1051 0 Nov 20 20:48 file2.txt

А теперь создадим в /tmp новый каталог 1, перейдем в него и сожмем zip_archive_test (никакие дополнительные опции, кроме совершенно необходимой -r, она же --recurse-paths не используются !!! тут будет сноска про то, что программа unzip, в отличие от zip, не поддерживает длинные опции, а только короткие !!! ): 

yura@Aspire-ES1-521 /tmp/1 $ sudo zip -r zip_archive_test.zip ../zip_archive_test/

Теперь проверим содержимое архива без его распаковки. Чтобы увидеть права доступа нужно задействовать переключатель -Z, причем она должна стоять первой среди прочих. 

yura@Aspire-ES1-521 /tmp/1 $ sudo unzip -Z -l zip_archive_test.zip
Archive:  zip_archive_test.zip
Zip file size: 938 bytes, number of entries: 5
drwxr-x---  3.0 unx        0 bx        0 stor 17-Nov-20 20:48 ../zip_archive_test/
drwsr-srwt  3.0 unx        0 bx        0 stor 17-Nov-20 20:48 ../zip_archive_test/dir/
-rw-r-----  3.0 unx        0 bx        0 stor 17-Nov-20 20:48 ../zip_archive_test/dir/file2.txt
-rw-r-----  3.0 unx        0 bx        0 stor 17-Nov-20 20:48 ../zip_archive_test/dir/file1.txt
-rwSr-S--t  3.0 unx        0 bx        0 stor 17-Nov-20 20:47 ../zip_archive_test/file.txt
5 files, 0 bytes uncompressed, 0 bytes compressed:  0.0%

Биты прав доступа никуда не делись, а владельцы исчезли (по крайней мере, в листинге их не видно). Распакуем архив (опять-таки, с правами суперпользователя): 

yura@Aspire-ES1-521 /tmp/2 $ sudo unzip ../1/zip_archive_test.zip

И получим: 

yura@Aspire-ES1-521 /tmp/2 $ sudo ls -l --almost-all --recursive zip_archive_test
zip_archive_test:
total 4
drwxr-xrwx 2 root root 4096 Nov 20 20:48 dir
-rw-r----x 1 root root    0 Nov 20 20:47 file.txt

zip_archive_test/dir:
total 0
-rw-r----- 1 root root 0 Nov 20 20:48 file1.txt
-rw-r----- 1 root root 0 Nov 20 20:48 file2.txt

Как видим, биты прав доступа (кроме них, отметки времени), остались. А вот владельцы пропали (вместо них всеми объектами владеет root), равно как и setuid, setguid и sticky биты.

Непонятно... В листинге архива мы видели и права доступа, и расширенные биты. А в распакованном виде мы видим только права доступа, причем расширенные биты пропали... Прежде, чем углубляться в дебри, заглянем в страницу руководства по zip. Там есть упоминание об опции -sc (--show-command):

Show the command line starting zip as processed and exit. The new command parser permutes the arguments, putting all options and any values associated with them before any non-option arguments. This allows an option to appear anywhere in the command line as long as any values that go with the option go with it. This option displays the command line as zip sees it, including any arguments from the environment such as from the ZIPOPT variable. Where allowed, options later in the command line can override options earlier in the command line.

Этот параметр поможет нам восстановить картину сжатия, т.е. убедиться, какие еще опции кроме -r были применены по-умолчанию (включая взятые из окружения командной строки): 

yura@Aspire-ES1-521 /tmp/3 $ sudo zip --show-command -r zip_archive_test.zip ../zip_archive_test/
command line:
'zip'  '--show-command'  '-r'  'zip_archive_test.zip'  '../zip_archive_test/'  

zip error: Interrupted (show command line)

Как видим, ничего особенного. Все та же -r, и больше ничего нового. Что ж, покурим man zip еще... И обнаружим параметр -X (--no-extra): !!! Тут будет сноска: Лично я прочитал все руководство к zip сверху вниз. И на нашел, кроме -X, ничего, что относилось бы к правам доступа и расширенным атрибутам Unix. !!!

Do not save extra file attributes (Extended Attributes on OS/2, uid/gid and file times on Unix). The zip format uses extra fields to include additional information for each entry. Some extra fields are specific to particular systems while others are applicable to all systems. Nor‐ mally when zip reads entries from an existing archive, it reads the extra fields it knows, strips the rest, and adds the extra fields applicable to that system. With -X, zip strips all old fields and only includes the Unicode and Zip64 extra fields (currently these two extra fields cannot be disabled).

Negating this option, -X-, includes all the default extra fields, but also copies over any unrecognized extra fields.

Do not save extra file attributes ..., далее Negating this option, -X-, includes all the default extra fields ... намекает на то, что по-умолчанию этот параметр включен. Даа, есть в чем поковыряться! Рассмотрим многострадальный файл file.txt. Мы видим, что биты доступа (включая расширенные) присутствуют в архиве в поле Unix file attributes. А вот владельцы в явном виде не встречаются. Но в примечании The central-directory extra field contains мы видим 

  - A subfield with ID 0x7875 (Unix UID/GID (any size)) and 11 data bytes:
    01 04 01 04 00 00 04 02 04 00 00.

Может именно эти загадочные циферки и есть наши владельцы? Надо разобраться. Как следует из раннего листинга каталога до упаковки, у файла file.txt (в десятичной системе счисления) UID=1025 и GID=1026. Ограничения на UID и GID в моей системе (LinuxMint 17.1) таковы (спасибо kirelagin за ответ: How big (in bits) is a Unix UID?): 

yura@Aspire-ES1-521 / $ grep 'UID' /usr/include/x86_64-linux-gnu/bits/typesizes.h
#define __UID_T_TYPE		__U32_TYPE

yura@Aspire-ES1-521 / $ grep 'GID' /usr/include/x86_64-linux-gnu/bits/typesizes.h
#define __GID_T_TYPE		__U32_TYPE


yura@Aspire-ES1-521 / $ grep '__U32_TYPE' /usr/include/x86_64-linux-gnu/bits/types.h 
#define __U32_TYPE		unsigned int

unsigned int - это 4 байта (а хоть бы и 2 байта - в ранних версиях Linux максимально возможное число пользователей было 65535, что укладывалось в 2 байта, по крайней мере в книжках так писали). Т.о. UID + GID = 8 байт (в крайнем случае, UID + GID = 4 байта). Но никак не 11 data bytes. Значит, UID и GID каким-то хитрым образом расположены (или может быть даже закодированы каким-нибудь эффективным способом, или разделены байтами-маркерами и т.п.) в строчке 01 04 01 04 00 00 04 02 04 00 00. Конечно, точный и окончательный ответ на этот вопрос даст исходный код утилиты zip. Но я исходники не смотрел. Я попробовал читать эти 11 байт задом наперед. Т.е. справа налево. Так, возьмем фрагмент 00 00 04 02, начиная с нулей (предположим, что здесь используется шестнадцатиричная система счисления). И переведем в десятичную: 00 00 04 0216=1026. Что такое 1026? Правильно, наш UID!

Проделаем то же самое для UID (опять таки, начнем с нулей): 00 00 04 01. И переведем в десятичную: 00 00 04 0116=102510. Что равняется UID.

Что там делают остальные байты? Непонятно... Надо смотреть исходный код.

Но вывод однозначен: при создании архива на Unix (по крайней мере, на Linux) с помощью zip с одной только опцией -r владелец, группа, биты прав доступа (рискну предположить, что Posix ACL и аттрибуты конкретной файловой системы, например аттрибут Immutable файловой системы ext4, сохранены в архиве не будут) сохраняются в ZIP-архиве.

Возникает вопрос, как эти аттрибуты достать. Это несложно. Чтобы извлечь биты SUID/SGID, нужно применить опцию -K:

[AtheOS, BeOS, Unix only] retain SUID/SGID/Tacky file attributes. Without this flag, these attribute bits are cleared for security reasons.

Попробуем: 

yura@Aspire-ES1-521 /tmp/4 $ sudo unzip -K ../1/zip_archive_test.zip
Затем 
yura@Aspire-ES1-521 /tmp/4 $ sudo ls -l --almost-all --recursive zip_archive_test
zip_archive_test:
total 4
drwsr-srwt 2 root root 4096 Nov 20 20:48 dir
-rwSr-S--t 1 root root    0 Nov 20 20:47 file.txt

zip_archive_test/dir:
total 0
-rw-r----- 1 root root 0 Nov 20 20:48 file1.txt
-rw-r----- 1 root root 0 Nov 20 20:48 file2.txt

Получилось!

Владельца/группу можно достать с помощь параметра -X:

[VMS, Unix, OS/2, NT, Tandem] restore owner/protection info (UICs and ACL entries) under VMS, or user and group info (UID/GID) under Unix, or access control lists (ACLs) under certain network-enabled versions of OS/2 (Warp Server with IBM LAN Server/Requester 3.0 to 5.0; Warp Connect with IBM Peer 1.0), or security ACLs under Windows NT. In most cases this will require special system privi‐ leges, and doubling the option (-XX) under NT instructs unzip to use privileges for extraction; but under Unix, for example, a user who belongs to several groups can restore files owned by any of those groups, as long as the user IDs match his or her own. Note that ordinary file attributes are always restored--this option applies only to optional, extra ownership info available on some operating systems. [NT's access control lists do not appear to be especially compatible with OS/2's, so no attempt is made at cross-platform portability of access privileges. It is not clear under what conditions this would ever be useful anyway.] 

yura@Aspire-ES1-521 /tmp/5 $ sudo unzip -X ../1/zip_archive_test.zip
И 
yura@Aspire-ES1-521 /tmp/5 $ sudo ls -l --almost-all --recursive zip_archive_test
zip_archive_test:
total 4
drwxr-xrwx 2 1050 1051 4096 Nov 20 20:48 dir
-rw-r----x 1 1025 1026    0 Nov 20 20:47 file.txt

zip_archive_test/dir:
total 0
-rw-r----- 1 1050 1051 0 Nov 20 20:48 file1.txt
-rw-r----- 1 1050 1051 0 Nov 20 20:48 file2.txt

Работает! Хотя раньше это был баг: #45 Problems with Unix UID/GID support in Zip/Unzip. Судя по статусу бага, он еще не исправлен разработчиками. Может мэйнтейнеры дистрибутива LinuxMint (Ubuntu, Debian) пропатчили программу сами?..

Что же все-таки не так с битами доступа?

Вообще-то, сохранение битов доступа в архиве - нужно. Если бы этой фичи не было в ZIP, он был бы почти бесполезным. Представьте, что вы скачали дистрибутив какой-нибудь программы в виде ZIP-архива. И в архиве не сохранены биты доступа. Придеться самостоятельно в таком случае искать исполняемые файлы и вручную делать их исполняемыми. Так что с этой точки зрения с программой unzip все в порядке. Проблема в другом. При распаковке архива unzip не обращает внимание на umask. Такое поведение по-умолчанию логично, если архив распаковывает суперпользователь (ведь он может назначить какие угодно права доступа). Но не для обычного пользователя.

Например, в моей системе у других пользователей нету никакого доступа к моим файлам, а именно: 

yura@Aspire-ES1-521 / $ umask
0027

Но unzip при распаковке все равно разрешает доступ другим пользователям наперекор umask: 

yura@Aspire-ES1-521 /tmp/6 $ unzip ../1/zip_archive_test.zip

yura@Aspire-ES1-521 /tmp/6 $ ls -l --almost-all --recursive zip_archive_test
zip_archive_test:
total 4
drwxr-xrwx 2 yura yura 4096 Nov 20 20:48 dir
-rw-r----x 1 yura yura    0 Nov 20 20:47 file.txt

zip_archive_test/dir:
total 0
-rw-r----- 1 yura yura 0 Nov 20 20:48 file1.txt
-rw-r----- 1 yura yura 0 Nov 20 20:48 file2.txt

И с этим ничего не поделаешь :( Как же круто было бы иметь в unzip опцию, подобную --no-same-permissions программы tar (я смотрел руководство к GNU Tar):

apply the user's umask when extracting permissions from the archive (default for ordinary users)

И все равно Info-ZIP рулит!

Не так-то просто исправлять баги в программах zip/unzip. И не так-то просто добавлять новые возможности. Почему? Казалось бы, нет ничего трудного в том, чтобы исправить или добавить. Ведь это - всего лишь утилиты командной строки, они не взаимодействуют с серверами в Интернете, не работают с базами данных и т.д. Но эти простые программы сохраняют совместимость (в некоторых случаях совместимости нет, об этом есть замечания в руководстве) со старыми программами и архивами, созданными сто лет назад. Эти утилиты компилируются под почти любую платформу (включая древние VMS и редкие в наши времена Amiga) и работают там. Админы их используют каждый день на серверах, и т.д. и т.п.

Так что если вы решили добавить новую опцию, которая применима только к битам доступа на Unix, то перед отправкой pull-request'а будьте добры проверить, что ваша новая фича работает во всех (по крайней мере, в нескольких) Unix'ах, а не только в вашем Linux любимом. И что опция игнорируется на других системах (но ведь мало кто взвалит на себя ношу проверки на той же Amiga, про MS-DOS и то все забыли). Сноски: [2] Все манипуляции с архивами я делаю от имени суперпользователя, т.е. с помощью sudo, чтобы чтобы предотвратить возможную нехватку прав на выполнение того или иного действия с содержимым архива.
