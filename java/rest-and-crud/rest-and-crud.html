Как начать изучать Java без помощи IDE

[2021, Декабрь]

Введение

REST расшифровывается как REpresentational State Transfer. Видимо, сначала был придуман благозвучный акроним, и лишь затем - более-менее подходящее значение. Как, к примеру, ELF - "Executable and Linkable Format" (или "Extensible Linking Format"), STRIDE - "Spoofing identity, Tampering with data, Repudiation threats, Information disclosure, Denial of service and Elevation of privileges".

На русском "наглядная передача состояния".

"Наглядная" потому что REST обходится без дополнительных сущностей, кроме изначально существовавших в протоколе HTTP.

"Передача" потому что без передачи между удаленными системами не обойтись.

"Состояния" потому REST - это надстройка над HTTP. А в протоколе HTTP состояния нет, так было задумано с самого начала. И вообще, любые взаимодействующие друг с другом программы по сути обмениваются своим состоянием. Только в обычных приложениях (например, консольное приложение на Java) это сложно заметить. Например, System.out.println("Some text"); печатает текст в стандартный поток вывода. Происходит это практически мгновенно, т.к. далеко ходить не надо - System.out находится совсем неподалеку (в адресном пространстве одной и той же JVM). А если бы System.out работал на удаленном сервере, стали бы очевидны следующие вещи:

  • удаленный System.out необязательно будет доступен в момент вызова println()
  • удаленный System.out необязательно напечатает текст мгновенно
  • удаленный System.out не может вернуть void, он должен сообщить о результате своей работы

Если экстраполировать, удаленные приложения не вызывают методы друг друга, а лишь сообщают о своих намерениях. Например, приложение А вызывает метод println() приложения B не напрямую, а примерно таким образом: А: Привет, B! У меня тут есть строка. Распечатай ее, пожалуйста. Принимай JSON. B: Здарова, А! Нет проблем. Побудь на проводе, щас сделаю. Напечатал. Возвращаю код HTTP 200.

У приложений, которые выполняются на разных компьютерах, нет ничего общего. Ни адресного пространства, ни переменных, ни объектов, ничего. Они не делят память, не сражаются за процессорное время, и живут каждое своей жизнью. Так что приложение А не может взять и приказать приложению Б что-то сделать. И наоборот. Можно сравнить их взаимодействие с расшаркивющимися друг перед другом аристократами. Но в отличие от дворянского этикета, в REST нет ничего лишнего.

За действия отвечают HTTP-методы: GET, POST, PUT, DELETE. И еще много других. Теоретически, набор HTTP-методов не ограничен. Т.к. метод - это всего лишь строка. Их можно придумать сколько угодно. Например, для CRUD можно внедрить соответствующие HTTP-методы: CREATE, READ, UPDATE, DELETE. Другое дело, что придется заставить других следовать такому же соглашению. А это непросто. Впрочем, поговорим о CRUD позже. А ведь когда-то было всего лишь два общепринятых метода: GET и POST. Тогда еще JavaScript не заполонил веб-странички. Когда нужно передать какие-то параметры, то в ход идет URL либо тело запроса. Зависит от конкретного API. В одних случаях достаточно передать ID в основной части URL (до знака "?"). Например, удаление пользователя по ID может осуществляться как запрос методом DELETE по адресу /api/v1/user/ид_пользователя. Для поиска пользователя подойдет GET-запрос вида /api/v1/user/ид?name=Alex&surname=Ivanov. Для сложных манипуляций передавать данные стоит в теле запроса. Ну там JSON или XML. На самом деле неважно, какой формат данных. Хоть BLOB (предварительно закодированный в base64). Так повелось, что JSON оказался наиболее удачным и созвучным JavaScript'у. Даже расшифровывается JSON как JavaScript Object Notation. Можно передавать и XML. Это не так важно. Важно, что в REST экономят на всем. Как стародавние времена конструкторы космических кораблей экономили массу аппаратов. Например, кодировка тела JSON запроса передается в HTTP-заголовке Content-type, а именно: Content-type: application/json; charset=utf-8 Методы REST не обязательно соответствуют методам CRUD. Кроме GET, POST, PUT, DELETE есть еще PATCH. Хотя их можно придумать сколько угодно, т.к. метод - это всего лишь текстовая строка. Сначала были только GET и POST. GET для запросов страничек, а POST для отправки форм. Раньше ведь не было REST API как таковых. Хотя у того же XML есть перед JSON ряд преимуществ (равно как и недостатков). Заголовки HTTP используются в основном в технических целях. Т.е. для указания кодировки тела запроса. Всякие access-токены и всё такое прочее.

В Java 9 структура каталогов другая, лень с ней разбираться. Да и зачем все эти новые технологии? Еще восьмую версию не полностью исследовали.

Установка

В одной из предыдущих заметок ( Еще раз об установке JDK) вскользь упоминалось о важности присутствия исходных текстов в каталоге с JDK.

Программы, которые устанавливаются в обход системы управления пакетами, не стоит распаковывать в домашний каталог. Конечно, для тех из них, которые обновляются автоматически, иного выбора нет (в противном случае придеться давать программе права суперпользователя на время обновления). В случае же с JDK автоматическое обновление не представляет затруднений, потому что его просто нет. Так же как и нет смысла перезатирать предыдущую версию. Мало ли, понадобиться запустить что-нибудь на конкретной версии. Тем более, что даже в распакованном виде JDK занимает сравнительно немного места. Так, версия JDK, упомянутая в предыдущем абзаце, в распакованном виде занимает на диске всего лишь 472 мегабайта, что даже меньше, чем объем оперативной памяти, потребляемый современными прикладными программами (привет, новый Skype for Linux).

Так вот, установка программ куда-нибудь в /opt, в первую очередь, более безопасна: никто, кроме root, не сможет подменить исполняемый файл программы. Хотя это и не нужно на самом деле: тот, кто может выполнять команды от имени обычного пользователя, без труда получит права суперпользователя, достаточно лишь подсунуть свою версию sudo. В общем, совсем не трудно замутить что-нибудь эдакое...

Чего действительно стоит опасаться, так это неумышленного повреждения файлов установленной программы обычным пользователем (в том числе скриптами, которые пишет пользователь для резервного копирования, например).

И что же делать с исходниками?

Ответ прост - читать :) Поскольку обычные пользователи имеют право только читать содержимое установленного нами JDK (а именно этого мы и добивались в предыдущем абзаце), можно смело открывать исходники в любимом текстовом редакторе, и, наслаждаясь подсветкой синтаксиса, неспешно созерцать их. Можно даже "редактировать", делать пометки, стирать ненужное (естественно, изменения сохранены не будут) и т.д. А поскольку файлы .java разложены в строгом соответствии со структурой пакетов, посмотреть исходный код класса, например, Integer совсем несложно будет [1]: 

$ less java/lang/Integer.java

find

Пусть мы знаем имя класса, например, ByteArrayOutputStream, но не знаем имени пакета, в котором он находится. То ли java.io, то ли java.nio... Текстовый редактор нам не поможет (мы же договорились не использовать IDE). Остается только искать. Это просто: 

$ find src -name 'ByteArrayOutputStream.java'

Другой пример. Пусть нас интересует содержимое некоторого пакета, но мы сами толком не знаем, что ищем. В этом случае для "обзора" пакетов можно вообще обойтись без утилит комадной строки! Лишь бы Bash (или другая оболочка) показывала содержимое каталога при нажатии на клавишу Tab. Можно путешествовать по дереву исходников, просматривать содержимое пакетов, возвращаться назад при необходимости (достаточно лишь стереть символы до последнего слэша).

Заключение

Конечно, учиться программировать без IDE придеться очень долго. Но главное - это начать. Совсем по-другому себя чувствуешь, когда знаешь, что ответы на основные вопросы всегда есть под рукой. Имена классов и пакетов запоминаются сами собой, выстраиваются в целостную систему. Не то что обрывки листингов, которые напрашиваются на copy-and-paste.
