Новое, клёвое и хреновое в Java 8. Часть 1. Потоки

[2017, Ноябрь]

Начало

В этой ламерской статье мы сосредоточимся на практическом применении Java 8 в энтэрпрайзных приложениях. Спасибо Benjamin Winterberg'у за офигенный туториал: Java 8 Stream Tutorial. Неудивительно, что его сайт находится на самой вершине выдачи Гугла.

Пара замечаний к исходному коду

К каждому примеру есть тест. Тесты нужны для того, чтобы убедиться, что разные подходы к решению одной и той же задачи выдают один и тот же результат. Кстати, код самих тестов написан в стиле Java 7. Это сделано намеренно, чтобы не отвлекать внимание от собственно примеров Java 8.

Помешательство на внутренних классах видно невооруженным глазом. Не знаю, как вам, но мне такой способ структурирования программы очень по душе: Внутренние классы в Java - классные! . Вообще, сегодня грань между кодом и данными постепенно стирается. Нет, это не отменяет концепции дедушки Никлауса Вирта ("Алгоритмы + Структуры данных = Программы"). Наоборот: сплетая алгоритмы в структуры, можно изобретать новые (или хотя бы упорядочивать существующие). Их можно пересылать, словно электронные письма, строить распределенные приложения, в которых между узлами передаются не данные, а код. Что-то подобное уже сделали: The Swarm Framework (про него я узнал в блоге Игоря Артамонова; на главной странице сказано, что сайт закрыт, но посты никуда не исчезли, и ссылки работают).

Также я использовал static import'ы везде, где только можно. Да, это нехорошо. Без IDE непонятно, что и откуда импортируется. Но именно короткие имена статически импортированных методов позволяют вкусить всю красоту кода на Java 8.

Реализация классов предметной области (называйте их как хотите: Entity, DTO и т.д.) - отдельная тема. Сейчас модно закладывать неизменяемость в такого рода классы. Т.е. методы, которые меняют состояние объекта, возвращают при каждом вызове новый объект. У меня немного не так. Я не пытался сделать объекты неизменяемыми. Я только избавился от традиционных getter'ов и setter'ов. По-моему, перегруженные методы, имена которых точь-в-точь повторяют имена соответствующих полей, намного круче. Они нагляднее. И помогают избежать лишней писанины, а именно: пижонские setter'ы справляются с созданием новых объектов не хуже, чем это делают builder'ы (т.е. могут инициализировать поля цепочкой, ключая вложенные). А может, и лучше: вместо немного сбивающих с толку фабричных методов вроде aCustomer() мы видим самые настоящие и прямолинейные конструкторы.

И еще: в тестах вместо заранее заготовленных наборов тестовых данных везде используются статические вспомогательные методы, которые при каждом вызове возвращают новый набор данных, т.е. новый объект, содержащий одни и те же данные. Зачем это нужно? Чтобы тесты (случайно) не модифицировали тестовые данные и не влияли таким образом друг на друга.

К делу!

Пожалуй, в каждом корпоративном приложении есть БД, в которой хранятся сведения о клиентах. Для внешнего мира эти сведени доступны через REST API. У нас есть для олицетворения клиента используется класс Customer. Данные о месте проживания клиента представлены в виде класса Address. Выглядят они так (в листинге 1 для краткости приведены только поля):

Листинг 1 - Классы предметной области

public class Customer {
    private String firstName;
    private String lastName;
    private Address address;
    private List phoneNumbers;
    
    ...
}

public class Address {
    private String country;
    private String town;
    private String street;
    private String buildingNumber;
    
    ...
}

Пусть нам дан список клиентов. Наша задача состоит в том, чтобы собрать все страны, в которых проживают клиенты, в некоторое множество. Попробуем сделать это в стиле Java 7 и получим код, приведенный в листинге 2. Он содержит все необходимое: проверки на null, обертывание результата неизменяемым множеством, рационально расставленные модификаторы final.

Листинг 2 - Стиль Java 7

public static Set<String> java7(final List<Customer> customers) {
    final Set<String> countries = new HashSet<>();
    for (final Customer customer : customers) {
        if (customer != null) {
            if (customer.address() != null) {
                String country = customer.address().country();
                countries.add(country);
            }
        }
    }
    return unmodifiableSet(countries);
}

Что же мы видим? Привычный цикл forEach и два вложенных if, кроме всего прочего. Попробуем теперь сделать то же самое, но с применением Java 8. В листинге 3 можно видеть "наивный" способ. По сути, от Java 8 там ничего нет, только новый цикл forEach

Листинг 3 - "Наивный" стиль Java 8

static Set<String> trivial(final List<Customer> customers) {
    final Set<String> countries = new HashSet<>();
    customers.forEach(customer -> {
        if (customer != null) {
            if (customer.address() != null) {
                String country = customer.address().country();
                countries.add(country);
            }
        }
    });
    return unmodifiableSet(countries);
}

Настоящая Java 8 - это потоки, т.е. streams. Это не потоки выполнения. И не потоки ввода-вывода. Что-то среднее. Ибо название "конвейер" им тоже не подходит: на конвейере разные операции над разными данными выполняются одновременно (хотя parallel streams уже имеют право называться конвейерами, как мне кажется; исходный код JDK я не смотрел, простите).

Листинг 4 - Настоящий стиль Java 8

static Set<String> stream(final List<Customer> customers) {
    return customers.
            stream().
            filter(Objects::nonNull).
            filter(customer -> nonNull(customer.address())).
            map(customer -> customer.address().country()).
            collect(collectingAndThen(toSet(), Collections::unmodifiableSet));
}

Что клёвого?

Код, написанный в стиле Java 8, намного проще читать. Никаких вложенных блоков, фигурных скобок и т.п. Вместо них - понятные имена методов, такие как filter(). Переменных нет вообще. Заботиться о предотвращении (случайного) изменения промежуточных результатов не нужно (равно как и добавлять модификатор final).

Форматирование исходного кода заметно упрощается (т.е. упрощаются правила для средств автоматического форматирования, таких как Eclipse Code Formatter). Вызовы соединяются в цепочку посредством точек, отступов нет (разве что внутри лямбда выражений; в любом случае, длинное лямбда-выражение лучше опредилить отдельно, т.е. сделать его самым обычным методом, а в filter(), к примеру, передать ссылку на красиво названный метод). В цепочку вызовов можно легко добавлять новые звенья - на форматирование существующего кода добавляемый никак не повлияет. Удалять код тоже проще - просто стираем строку, никакие фигурные скобки от этого не сдвинутся. И еще автодополнение IDE работает всегда, на каждом этапе, и с гораздо большей эффективностью (и не нужно мозговать над именем очередного вспомогательного метода, предлагаемых идиом более чем достаточно).

Что хренового?

Стэктрэйсы

Об этом уже писал Tal Weiss в статье The Dark Side Of Lambda Expressions in Java 8. От себя могу добавить, что увлекаться лямбда-выражениями не стоит еще и потому, что они безымянные. На них нельзя "повесить" аспекты. В принципе, это возможно, т.е. можно нагородить костылей. Ведь на самом деле лямбда-функции - это синтетические методы, которые имеют во время выполнения имена вроде *lambda$*.

С потоками ситуация еще хуже. Добавим в метод country() класса Address код "искусственной" ошибки: 

if (Objects.equals(country, "USA")) {
    throw new IllegalStateException("This is workaround for demonstrating nasty Java 8 stacktraces.");
}

Отсутствие переменных

Да, промежуточные результаты сохраняются во внуренних переменных (т.е. полях) объектов потоков. Но чтобы их посмотреть, надо поставить точку останова внутри исходного кода JDK. Какие имена будут у этих переменных? Правильно, они будут называться так, как это реализовано в JDK, т.е. без какой-либо связи с предметной областью приложения.

Логгирование

Проблема проистекает из видимого преимущества цепочки вызовов. Например, потребовалось нам записать промежуточный результат в журнал. Как это сделать? Для этого есть метод peek() . В документации сказано, что он служит целям отладки:

This method exists mainly to support debugging, where you want to see the elements as they flow past a certain point in a pipeline:
. Но ежу понятно, что этот метод может использоваться и для других целей. В принципе, можно любой API использовать не по назначению. Но имя peek() слишком общее. Почему бы не назвать "отладочный" метод debug() или trace или log? Непонятно...

А еще прикол в том, что метод peek() возвращает поток. Т.е. чтобы просто записать в лог промежуточный результат (т.е. записать его целиком: элементы разделить запятыми, по краям поставить скобки и т.д.) придеться этот поток предварительно преобразовать в список. Хотя это уже неактуально: наверняка, в современных версиях библиотек, таких как Log4j, можно логгировать потоки целиком.

Что нового?

Ничего, кроме лямбда-выражений и ссылок на методы. Потоки можно реализовать и на Java 7. Но придеться повозиться с типами. В случае с лямбда-выражениями компилятор сам определяет тип возвращаемого значение. А вот в Java 7 придеться указывать тип явно. Что-то в таком роде (я не проверял, это лишь предположение): 

...	
new MyLambda implements Lambda {
    @Override
    public boolean filter(final Object input, final Class returnType) {
        ...
    }
}
...

А вот ссылки на методы - это реально новая фича. До нее в Java можно было использовать указатели на методы. Но работали указатели через механизм рефлексии, т.е. только во время выполнения. Имя требуемого метода надо было задавать как строку (т.е. как строковый литерал, константу, переменную), проверок типов не было, автодополнения IDE - подавно.

Потоки Java 8 выглядят магически. Но это не так. Чтобы в этом убедиться, не нужно смотреть исходники JDK. Как пишет Benjamin Winterberg в своей статье:

An important characteristic of intermediate operations is laziness.
И еще:
The order of the result might be surprising. A naive approach would be to execute the operations horizontally one after another on all elements of the stream. But instead each element moves along the chain vertically. The first string "d2" passes filter then forEach, only then the second string "a2" is processed.

Выполнение операций над элементами потока "горизонтально" и в самом деле было бы неэффективно. Но почему же тогда потоки "ленивые"?

Это не гениальный замысел разработчиков Java 8. Просто что в Java, после того, как метод вернул значение или выполнился без возвращения значения (возвращаемый тип void или в методе возникло исклчение), в него нельзя вернуться. Когда методы выполяются в цепочке, звенья ничего не делают, а только "запоминают" те операции, которые должны будут выполниться позднее. Скажем, метод filter() всего лишь сохраняет где-то в приватных полях потока ссылку на лямбда-выражение (по сути, на экземпляр анонимного класса) или ссылку на метод. Терминальная операция (такая как forEach() и т.д.) просто вызывает записанные ранее действия для каждого из элементов в потоке (как уже было сказано выше, не все элементы "доживают" до терминальной операции).

Так что потоки Java 8 в принципе не могут работать с коллекциями, количество элементов в которых неизвестно заранее. Например, настоящий диапазон (в котором каждый следующий элемент последовательности вычисляется на лету, а не хранится в памяти), как это делает функция range() в Python 3.x, в Java 8 отсутствует.
